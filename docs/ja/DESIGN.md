# auto-oss 設計

このドキュメントは auto-oss のアーキテクチャと、各判断の理由の記録。記述的な文書であり、
規範となるプロトコルは [SPEC.md](SPEC.md) (英語版が正)。

## 問題

コーディングエージェントがパッチを安価にした代償を、メンテナが払っている。
無差別な AI 生成 PR・バグレポートは OSS プロジェクトの公然の負担になった。
一方で、Issue → PR ループの既存自動化 — OpenHands Resolver、GitHub Copilot
coding agent、Sentry Seer — はすべて**オーナー側**の自動化で、リポジトリの
チーム自身が自分のバックログにエージェントを組み込むものだった。

逆向きは誰も設計していなかった。実際に問題を踏んだ**外部の利用者**にも
エージェントはいる。auto-oss は、その利用者のエージェントがフィードバックを
パッチにして、**メンテナが設定した条件に照らして事前に screening 済みで、
機械的にそれと識別できる形で** upstream に届けるために存在する。これは
「信頼できる」より意図的に弱い主張である: 準拠は検証可能(`autos verify` は
実際に再導出できるもの——GitHub 自身から得る diff サイズ——を申告の言い分
どおりに信じるのではなく再計算する)だが、ここには貢献者の誠実さを検証する
仕組みはなく、ゲート結果自体もまだ自己申告のままである。
[セキュリティモデル](#セキュリティモデル)を参照。

## 立ち位置: ツールではなくプロトコル

難しいのはパッチを作ることではない(それはエージェントがやる)。難しいのは、
頼まれていないエージェント PR がスパムと区別できないこと。だから auto-oss の
中核は、ファイルフォーマットとして固定された契約である:

1. **メンテナの opt-in** — `auto-oss.yml` を公開したリポジトリだけが提出を
   受ける。ファイルの不在は拒否であり、準拠クライアントはそれに従う。
2. **品質ゲート** — パッチが通るべきコマンド(test / lint / build)を policy が
   宣言する。不合格のパッチは構造化 Issue に格下げされ、PR にはならない。
3. **開示と責任** — すべての提出に機械可読な来歴ブロック(元フィードバック・
   backend・ゲート結果)が埋め込まれ、提出者は常に人間である。

パッチ生成は既存エージェント(現在は Claude Code、差し替え可能)に意図的に
委譲する。auto-oss の価値は契約であって、もう一つのエージェントではない。
そして契約は逆向きにも機能する: policy の公開は「これ以外は受け入れない」の
表明でもある。

## アーキテクチャ

```
利用者                          autos CLI                          upstream リポジトリ
 |                                 |                                    |
 | 「この挙動がおかしい」          |                                    |
 |-------------------------------->|                                    |
 |                                 |--- policy 取得 (auto-oss.yml) ---->|
 |                                 |<-- 受け入れ条件 -------------------|
 |                                 |                                    |
 |                                 |-- 新規 workdir に clone            |
 |                                 |-- backend がパッチ生成             |
 |                                 |-- 宣言されたゲートを実行           |
 |                                 |                                    |
 |<-- diff + ゲート結果 -----------|                                    |
 | 承認                            |                                    |
 |-------------------------------->|-- ブランチ push、PR 作成 --------->|
 |                                 |   (権限があれば直接、なければ fork)|
 |                                 |                                    |
 |                                 |   ゲート不通過 → 構造化 Issue
```

参照実装(Rust、単一バイナリ)のモジュール構成:

| モジュール | 責務 | SPEC 対応 |
|---|---|---|
| `policy` | 発見・パース・デフォルト・リポジトリ参照 | §1–2 |
| `fix` | 提出パイプライン | §4 |
| `backend` | エージェント抽象(`claude-code` / `human`) | — |
| `gates` | ゲート実行 | §2 |
| `metadata` | メタデータブロック生成 | §3 |
| `verify` | 受け側の準拠検証 | §3–5 |

## 判断

### policy ファイルはリポジトリのルート

探索順は `auto-oss.yml` → `.github/auto-oss.yml`。ルート配置を推奨する。
opt-in は契約であり、契約は見えることに価値がある(`FUNDING.yml` が
`.github/` に隠れているのはメタデータだから。これは違う)。

### メタデータは HTML コメント内の YAML

提出ブロックは PR / Issue 本文の `<!-- auto-oss:v0 ... -->` に載る。人間の
読者には見えず、メンテナのツールからは機械可読で、そして重要なのは移植性:
フォージ固有の機能に依存しないため、git と PR があるところならどこでも
プロトコルが成立する。

### backend は差し替え可能、クライアントは LLM を呼ばない

`autos` はプロンプト(フィードバック + policy の制約)を組み立てて backend の
サブプロセスに委譲する。クライアントは小さく保たれ、モデルの選択は利用者に
残り、コーディングエージェントの進歩は全部ただで継承される。予約 backend
`human` が存在するのは開示が正直であるべきだから: 手書きのパッチはエージェント
関与を装うのではなく、手書きだと名乗る — 機構(opt-in・ゲート・来歴)は同じに
適用される。

### 提出前の人間の確認

CLI は必ず最終 diff・ゲート結果・提出本文を提示し、明示的な承認を待つ。
policy の `require.human_review` はメタデータの宣誓(`human_reviewed`)に
対応する。v0 では宣誓を技術的に検証できない — その価値は契約的なもので、
虚偽の宣誓は拒否と ban の文書化された根拠になる。

### PR を強行せず格下げする

ゲート不通過や diff 超過のとき、実行結果は捨てられない: 再現手順・環境・
部分 diff には価値があるので、policy の `fallback` が許すなら構造化 Issue と
して提出する。「品質が証明できたときだけ PR」がスパム対策の核心。

### 権限があれば直接 push、なければ fork

PR の head ブランチはフォージ上に存在する必要がある。push 権限のある貢献者は
upstream 本体にブランチを切り、外部の人は fork を経由する — GitHub が外部に
提供する唯一の経路。両経路とも公開前に実リポジトリで検証済み。

### Rust、単一バイナリ

クライアントは貢献者のマシンで動く。依存の少ない単一バイナリは配布に向く。
パッチ生成は委譲されるので、クライアント本体はオーケストレーションと I/O のみ。

### 名前: プロトコルは auto-oss、コマンドは autos

`autos` はギリシャ語 **αὐτός** — *自分自身・自らの意思で*。それがこの
プロトコルの要点: 問題を感じた本人が、自分のエージェントを通して自ら動く。
プロジェクト名として "autos" は自動車の検索結果に沈むので、検索される名前は
`auto-oss` のまま、語根はバイナリに残した(ripgrep → `rg` の慣行)。

## セキュリティモデル

プロトコルはメンテナを望まぬ提出から守る。逆方向は [SECURITY.md](../SECURITY.md)
に記載: policy のゲートは**貢献者の**マシンで実行されるため、`autos fix` の
実行は相手リポジトリを信頼する行為であり、リポジトリの内容は backend
エージェントへの信頼できない入力(prompt injection)である。ゲートの
サンドボックス実行は v1 候補。

## 将来(SPEC v1 候補)

- `accepts.paths` — パスの許可/禁止リスト。公開後に気づいた: `docs` scope は
  プロジェクトの判断ログまで書き換えの対象にしてしまう。
- 両側でのゲートのサンドボックス実行: 貢献者のマシンは対象リポジトリが
  宣言したコマンドを実行するし(セキュリティモデル参照)、申告を検証する
  ためにサーバー側でゲートを再実行したいリポジトリにも、隔離された実行
  場所がない。
- backend の追加(OpenHands、OpenCode)。
- 受け側でのレートリミット強制 — `limits.per_author_per_week` は現状
  クライアント側の自主遵守(SPEC の SHOULD)のみで、クライアントが正直に
  申告しているかをリポジトリ側で検証する手段はない。
- 独立に再導出できないメタデータ申告(`human_reviewed`、ゲート合格結果)への
  署名または評判システム。`autos verify` はすでに diff サイズを GitHub から
  直接再導出しており、提出側の自己申告の数字を鵜呑みにしない。ゲート合格・
  レビューの申告そのものには、リポジトリ自身の CI が同じコマンドを再実行する
  以外の検証手段がない。
- GitHub 以外のフォージ(GitLab、Codeberg)— SPEC はすでにフォージ非依存で、
  `gh` に縛られているのはクライアントだけ。

## 継続するリスク

- **プラットフォームによる取り込み。** GitHub 等が「外部エージェント PR」の
  公式制御を実装するかもしれない。オープンでフォージ非依存なデファクト契約を
  先に作ることが防御 — オープン標準の取り込まれは勝ち方の一つでもある。
- **虚偽メタデータ。** `human_reviewed` やゲート合格の申告は偽装できる。
  `autos verify` は形式を検証し diff サイズを独立に再導出するが、宣言された
  ゲート自体は実行しない——リポジトリが自身の CI に policy と同じコマンドを
  独立に走らせること(このリポジトリの `ci.yml` がそうしている)が実際の
  チェックになる。それをしていない opt-in リポジトリは、ゲートの申告を
  そのまま信用していることになる。
- **コールドスタート。** opt-in リポジトリのないプロトコルはただの仕様書。
  対策: 初日からのドッグフーディングと、エージェントスパムに公然と困っている
  メンテナへの提案 — 彼らにとって policy ファイルは防波堤を兼ねる。
