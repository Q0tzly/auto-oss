# セキュリティ

## 脆弱性の報告

このリポジトリの GitHub private vulnerability reporting(Security → Report a
vulnerability)を使うこと。セキュリティ上の問題を公開 issue に書かないこと。

## `autos` 利用者の脅威モデル

プロトコルはメンテナを望まぬ提出から守る。逆方向も明記しておく必要がある:
**`autos` の提出系コマンド(`fix`・`feat`・`docs`・`refactor`・`test`・
`typo`・`resume`)のいずれかをリポジトリに対して実行することは、そのリポジトリ
を信頼する行為である**。理由は:

- policy の `gates.*` コマンドは**あなたの**マシン上、clone の中で実行され
  る。悪意ある opt-in リポジトリは悪意あるゲートを宣言できる。
- backend エージェントは編集権限付きで clone されたコードに対して動作する。
  リポジトリの内容は、エージェントへの信頼できない入力(prompt injection)
  である。

現在の緩和策: これらのコマンドはどれも、clone の中でゲートを実行する直前に
全コマンドを表示し、明示的な確認を求める。提出も、別途の明示的な確認なしには
行われない。ただし、確認を求める時点ですでに clone とパッチ生成(エージェントに
よるリポジトリ内容の読み込み)は完了している——この確認が守っているのは
ゲート実行の一線であって、リポジトリ内容への曝露そのものではない。
ゲートのサンドボックス実行は SPEC v1 候補。それまでは、これらのどのコマンド
も、そのリポジトリからの `git clone && make` と同じ警戒度で扱うこと。
